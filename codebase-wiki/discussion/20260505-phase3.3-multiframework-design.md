## Overview

Phase 3.3 extends the Telegraph Agent Runtime to support multiple execution frameworks beyond pi-ai. After the 2026-05-20 alignment, framework adapters should sit under the Native Harness or External Agent Runtime boundary rather than becoming product-level backends by themselves.

---

## Adapter Pattern

### Interface Contract

All runtime adapters must implement `RuntimeExecutor`:

```typescript
export interface RuntimeExecutor {
  readonly id: string;
  readonly label: string;
  run(input: RuntimeInput): AsyncIterable<RuntimeEvent>;
}
```

### Implementation Strategy

1. **Framework-Specific Classes**: Each framework gets a dedicated `*Runtime` class
   - `PiAiRuntime` (existing)
   - `EmbeddedExecutionKernel` (Native Harness bottom layer)
   - `LangGraphRuntime` (Phase 3.3)
   - `VercelAiRuntime` (Phase 3.3)
   - `MastraRuntime` (Phase 3.3+)

2. **Factory Pattern**: `createRuntime(settings)` dispatches by `backend` type
   ```typescript
   export function createRuntime(settings: AgentRuntimeSettings): RuntimeExecutor {
     switch (settings.backend) {
       case 'pi-ai':
         return createPiAiRuntime(settings);
      case 'native-embedded':
        return createEmbeddedExecutionKernel(settings);
       case 'langgraph':
         return createLangGraphRuntime(settings);
       case 'vercel-ai':
         return createVercelAiRuntime(settings);
       default:
         throw new Error(`Unknown backend: ${settings.backend}`);
     }
   }
   ```

3. **Event Normalization**: All frameworks emit the same `RuntimeEvent` schema
   - `run_started`, `run_completed`, `run_failed`, `run_cancelled` (lifecycle)
   - `model_event`, `assistant_message`, `assistant_delta` (model)
   - `tool_call`, `tool_result`, `tool_error` (tools)
   - `step_started`, `step_completed` (workflow/graph steps)

## Framework-Specific Integrations

### LangGraphRuntime

**Dependencies**: `@langchain/langgraph`

**State Machine Pattern**:
- LangGraph executes as state transitions
- Each state update represents a step (message, tool call, etc.)
- Terminal states trigger `run_completed` or `run_failed`

**Event Mapping**:
- LangGraph `messages` node → `model_event` or `assistant_message`
- LangGraph `tools` node → `tool_call` + `tool_result` 
- LangGraph state edges → `edge_taken` (workflow event)

**Example**:
```typescript
interface LangGraphSettings {
  graphConfig: StateGraph;  // User-defined state machine
  modelName: string;
  timeout?: number;
  maxIterations?: number;
}

class LangGraphRuntime extends BaseAgentRuntime {
  async *run(input: RuntimeInput): AsyncIterable<RuntimeEvent> {
    // Initialize state from input.message
    // Stream graph.stream() and convert events
    // Detect tool calls and execute
    // Emit RuntimeEvents for all state transitions
  }
}
```

### VercelAiRuntime

**Dependencies**: `ai` (Vercel AI SDK)

**Streaming Pattern**:
- Vercel AI SDK provides `generateText` and `streamText` methods
- Returns `ContentBlock[]` with assistant messages and tool uses
- Native tool execution support via `tools` parameter

**Event Mapping**:
- `streamText` text deltas → `assistant_delta`
- Tool uses in response → `tool_call`
- Tool results → `tool_result`
- Final message → `assistant_message`

**Example**:
```typescript
interface VercelAiSettings {
  model: LanguageModel;  // e.g., openai('gpt-4-turbo')
  tools?: { [key: string]: Tool };
  systemPrompt?: string;
  temperature?: number;
}

class VercelAiRuntime extends BaseAgentRuntime {
  async *run(input: RuntimeInput): AsyncIterable<RuntimeEvent> {
    // Create messages array from session history
    // Call streamText with tools
    // For each chunk: emit assistant_delta
    // For each tool_use: emit tool_call, execute, emit tool_result
    // Finally: emit assistant_message with full response
  }
}
```

### MastraRuntime

**Dependencies**: `@mastraai/core`

**Agent Pattern**:
- Mastra has `Agent` class for agentic loops
- `run()` method handles multi-turn with tool use
- Returns `AgentRunResult` with message and tool calls

**Event Mapping**:
- `Agent.run()` iteration → step events
- Tool calls detected → `tool_call` events
- Tool execution → `tool_result` events
- Final response → `assistant_message`

## Type System Alignment

### AgentBackendKind Extension

```typescript
export type AgentBackendKind =
  | 'pi-ai'           // Streaming, no tools
  | 'native-embedded' // Native Harness embedded kernel
  | 'langgraph'       // LangGraph state machine
  | 'vercel-ai'       // Vercel AI SDK
  | 'mastra';         // Mastra agents
```

```typescript
export interface RuntimeSettings {
  backend: AgentBackendKind;
  // Framework-specific config (union type)

}
```

## Event Normalization Strategy

### Challenges

1. **Schema Mismatch**: Different frameworks return different event shapes
2. **Timestamp Format**: Some frameworks use ISO strings, others milliseconds
3. **Tool Representation**: Different formats for tool calls and results
4. **Error Handling**: Varying error structures

### Solution: Event Adapter Layer

Each runtime has internal methods to convert framework events to `RuntimeEvent`:

```typescript
// Internal helper in each *Runtime class
private normalizeToRuntimeEvent(frameworkEvent: any): RuntimeEvent {
  return {
    schemaVersion: RUNTIME_CONTRACT_SCHEMA_VERSION,
    type: frameworkEvent.eventType,  // mapped
    ts: Date.now(),
    // ... other required fields
  };
}
```

### Tool Call Normalization

All runtimes use shared `ToolCallParser` to detect and standardize tool calls:

```typescript
// Works across frameworks (XML, JSON, OpenAI function format)
const calls = ToolCallParser.parseToolCalls(llmResponse);
// Returns: ParsedToolCall[] with id, toolId, input
```

## Known Limitations & Future Work

1. **Framework Dependencies**: Each runtime requires its framework installed
   - Solution: Lazy loading, optional peer dependencies

2. **Feature Parity**: Not all frameworks support all features (e.g., sub-agents)
   - Solution: Document feature matrix, skip unsupported on request

3. **Version Compatibility**: Framework APIs change
   - Solution: Version compatibility matrix, adapter updates

4. **Performance**: Different frameworks have different performance profiles
   - Solution: Benchmarking suite, selectivity by use case
